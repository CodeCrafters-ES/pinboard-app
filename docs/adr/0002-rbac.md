# ADR-002 — Control de acceso basado en roles (RBAC) + RLS

**Estado:** Aceptado  
**Fecha:** 2026-06-17  
**Autores:** Alex Zapata  
**Issues:** [EPIC-S00 #28](https://github.com/CodeCrafters-ES/pinboard-app/issues/28) · [I-F-S00-03-01 #36](https://github.com/CodeCrafters-ES/pinboard-app/issues/36) · [I-F-A00-02-01 #53](https://github.com/CodeCrafters-ES/pinboard-app/issues/53) · [I-F-A00-02-02 #54](https://github.com/CodeCrafters-ES/pinboard-app/issues/54)

---

## Contexto

El proyecto necesita un modelo de autorización que:

- Cubra una app móvil con tres categorías de usuarios (staff, mandos intermedios y administradores).
- Garantice que ningún cliente (React Native / SDK de Supabase) pueda saltarse las reglas por error, aunque las claves de acceso queden expuestas.
- Sea auditable y versionado como parte del código (no como configuración en un panel de control).
- Permita escalar las reglas sin reescribir el cliente.

Decisiones previas que condicionan este ADR:

- **Base de datos:** Supabase Postgres 15 (Row Level Security nativa).
- **Autenticación:** Supabase Auth — `auth.uid()` es el identificador de sesión de confianza.
- **Helpers SECURITY DEFINER:** definidos en la migración `0003` (ver `supabase/migrations/20260617190000_security_definer_role_helpers.sql`).

---

## Decisión

### Roles

Tres roles fijos en el MVP con jerarquía inclusiva `admin > manager > staff`:

| Rol | Descripción |
|---|---|
| `admin` | Superusuario funcional: gestión de personas, contenidos críticos y configuraciones. |
| `manager` | Liderazgo de equipo: analytics agregado, contenidos propios, moderación limitada. |
| `staff` | Usuario base: feed, reacciones, valoraciones, comentarios, chats, eventos. |

### Fuente de verdad

El rol efectivo se almacena **únicamente** en `public.profiles.role` (enum `user_role`). No se replica en JWT claims. Cada request lo resuelve en base de datos.

### Lectura del rol en RLS

`auth_role()` (`SECURITY DEFINER`, `search_path = public, auth`) es el único punto de lectura de `profiles.role` para las policies. Sus wrappers `is_admin()`, `is_manager()` e `is_staff()` son también `SECURITY DEFINER` con `search_path` fijado (migración `0017`). Los cuatro helpers tienen `EXECUTE` revocado a `PUBLIC` y concedido solo a `authenticated` y `service_role`.

```sql
-- Firmas canónicas — estado final tras migraciones 0003 + fix_auth_role + 0017
CREATE OR REPLACE FUNCTION public.auth_role()
RETURNS public.user_role
LANGUAGE sql STABLE SECURITY DEFINER
SET search_path = public, auth
AS $$ SELECT role FROM public.profiles WHERE user_id = auth.uid(); $$;
-- Devuelve el rol del usuario autenticado; NULL si no hay sesión.

CREATE OR REPLACE FUNCTION public.is_admin()
RETURNS boolean
LANGUAGE sql STABLE SECURITY DEFINER
SET search_path = public, auth
AS $$ SELECT COALESCE(public.auth_role() = 'admin', false); $$;
-- True solo para admin. Nunca devuelve NULL.

CREATE OR REPLACE FUNCTION public.is_manager()
RETURNS boolean
LANGUAGE sql STABLE SECURITY DEFINER
SET search_path = public, auth
AS $$ SELECT COALESCE(public.auth_role() IN ('admin', 'manager'), false); $$;
-- True para admin y manager (jerarquía inclusiva). Nunca devuelve NULL.

CREATE OR REPLACE FUNCTION public.is_staff()
RETURNS boolean
LANGUAGE sql STABLE SECURITY DEFINER
SET search_path = public, auth
AS $$ SELECT COALESCE(public.auth_role() IN ('admin', 'manager', 'staff'), false); $$;
-- True para cualquier rol autenticado. Nunca devuelve NULL.
```

### Autorización

La autorización **vive en las RLS policies de Postgres**, nunca en el cliente. El cliente puede ocultar elementos de UI, pero si la policy falla la query falla. Todo cambio de permisos exige:

1. Actualizar la matriz en este ADR.
2. Nueva migración SQL en `supabase/migrations/`.
3. Tests pgTAP (≥ 1 positivo + 1 negativo por policy) en `supabase/tests/rls/`.
4. PR con los tres entregables juntos.

## RLS Policies — SQL canónico

Expresiones exactas implementadas en las migraciones de `supabase/migrations/`. Las operaciones sin policy (INSERT/UPDATE/DELETE en `engagement_sessions`) son denegación implícita por RLS.

### `profiles`

| Policy | Op | USING | WITH CHECK |
|---|---|---|---|
| `profiles_select_authenticated` | SELECT | `true` | — |
| `profiles_insert_admin` | INSERT | — | `is_admin()` |
| `profiles_update_self_or_admin` | UPDATE | `user_id = auth.uid() or is_admin()` | `(user_id = auth.uid() and role = auth_role()) or is_admin()` |
| `profiles_delete_admin` | DELETE | `is_admin()` | — |

> `WITH CHECK` en UPDATE impide que un no-admin cambie su propio `role`. `auth_role()` lee el rol actual del llamante sin recursión gracias a `SECURITY DEFINER`.

### `posts`

| Policy | Op | USING | WITH CHECK |
|---|---|---|---|
| `posts_select_authenticated` | SELECT | `true` | — |
| `posts_insert_manager_or_admin` | INSERT | — | `is_manager() and author_id = auth.uid()` |
| `posts_update_own_or_admin` | UPDATE | `is_admin() or (is_manager() and author_id = auth.uid())` | `is_admin() or (is_manager() and author_id = auth.uid())` |
| `posts_delete_own_or_admin` | DELETE | `is_admin() or (is_manager() and author_id = auth.uid())` | — |

### `post_reactions`

| Policy | Op | USING | WITH CHECK |
|---|---|---|---|
| `post_reactions_select_authenticated` | SELECT | `true` | — |
| `post_reactions_insert_self` | INSERT | — | `user_id = auth.uid()` |
| `post_reactions_update_self` | UPDATE | `user_id = auth.uid()` | `user_id = auth.uid()` |
| `post_reactions_delete_self_or_admin` | DELETE | `user_id = auth.uid() or is_admin()` | — |

### `post_ratings`

| Policy | Op | USING | WITH CHECK |
|---|---|---|---|
| `post_ratings_select_authenticated` | SELECT | `true` | — |
| `post_ratings_insert_self` | INSERT | — | `user_id = auth.uid()` |
| `post_ratings_update_self` | UPDATE | `user_id = auth.uid()` | `user_id = auth.uid()` |

> Sin policy DELETE por diseño: las valoraciones son permanentes y actualizables (UPSERT).

### `post_comments`

| Policy | Op | USING | WITH CHECK |
|---|---|---|---|
| `post_comments_select_authenticated` | SELECT | `true` | — |
| `post_comments_insert_self` | INSERT | — | `author_id = auth.uid()` |

## Matriz de permisos — Storage

Los buckets siguen la convención de path `{user_id}/...` para que las policies validen la propiedad sin necesidad de un JOIN.

| Bucket | Admin | Manager | Staff |
|---|---|---|---|
| `avatars` | Upload all / read all | Upload own / read all | Upload own / read all |
| `post-images` | Upload + delete all | Upload + delete own | Read only |
| `event-images` | Upload + delete all | Upload + delete own | Read only |

### Storage Policies — SQL canónico

Todas las policies operan sobre `storage.objects`. La convención de path `{user_id}/...` permite validar propiedad con `(storage.foldername(name))[1]` sin necesidad de JOIN.

#### Bucket `avatars` (público · límite 2 MB · solo `image/webp`)

| Policy | Op | USING | WITH CHECK |
|---|---|---|---|
| `avatars_select_authenticated` | SELECT | `bucket_id = 'avatars'` | — |
| `avatars_insert_own` | INSERT | — | `bucket_id = 'avatars' AND auth.uid()::text = (storage.foldername(name))[1]` |
| `avatars_update_own` | UPDATE | `bucket_id = 'avatars' AND auth.uid()::text = (storage.foldername(name))[1]` | `bucket_id = 'avatars' AND auth.uid()::text = (storage.foldername(name))[1]` |

> Bucket público: cualquier URL del objeto es accesible sin firma. Sin policy DELETE; los avatares se reemplazan vía UPDATE.

#### Buckets `post-images` y `event-images` (privados · límite 5 MB · `image/webp`)

Implementadas en `20260708000000_rls_storage_post_event_images_n02_03_03.sql` (#150). La policy UPDATE soporta el re-upload (`upsert`) sobre el mismo path.

| Policy | Op | USING | WITH CHECK |
|---|---|---|---|
| `post_images_select_authenticated` | SELECT | `bucket_id = 'post-images'` | — |
| `post_images_insert_manager_or_admin` | INSERT | — | `bucket_id = 'post-images' AND (is_admin() OR (is_manager() AND auth.uid()::text = (storage.foldername(name))[1]))` |
| `post_images_update_own_or_admin` | UPDATE | `bucket_id = 'post-images' AND (is_admin() OR auth.uid()::text = (storage.foldername(name))[1])` | `bucket_id = 'post-images' AND (is_admin() OR (is_manager() AND auth.uid()::text = (storage.foldername(name))[1]))` |
| `post_images_delete_own_or_admin` | DELETE | `bucket_id = 'post-images' AND (is_admin() OR auth.uid()::text = (storage.foldername(name))[1])` | — |

> Las mismas expresiones aplican a `event-images` cambiando `bucket_id`. `staff` solo tiene SELECT; las policies de escritura requieren `is_manager()` implícito (admin) o propiedad de carpeta (manager).

## Auditoría de cambios de rol

Cualquier cambio en `profiles.role` queda registrado automáticamente en `public.role_audit` mediante el trigger `profiles_role_audit` (función `log_role_change()`, `SECURITY DEFINER`).

| Columna | Descripción |
|---|---|
| `target_user_id` | FK → `profiles(id)` del usuario cuyo rol cambió (`SET NULL` si el perfil se elimina) |
| `changed_by` | `auth.uid()` del actor; `NULL` si el cambio vino de `service_role` |
| `from_role` | Rol anterior (`NULL` en la primera asignación) |
| `to_role` | Nuevo rol |
| `changed_at` | Timestamp del cambio (`DEFAULT now()`) |

**Políticas:**

- Solo `admin` puede `SELECT`.
- No existen policies de INSERT/UPDATE/DELETE para clientes. RLS deniega por defecto, reforzado con `REVOKE`.
- El trigger escribe directamente como `postgres` (SECURITY DEFINER), sin pasar por RLS.

**Retención:** mínimo 2 años desde `changed_at` (ver decisiones de TTL en EPIC-S00).

## Consecuencias

**Positivas:**

- Modelo simple y auditable; toda la autorización en un único lugar (Postgres).
- Un cliente comprometido o mal configurado no puede elevar privilegios.
- Cambios de permisos siempre dejan rastro en el historial de migraciones.

**Negativas / limitaciones conocidas:**

- Jerarquía de roles plana: no hay permisos granulares por recurso individual (ACL fila a fila más allá de RBAC).
- Si el proyecto crece a multi-tenant, este ADR necesita revisión completa (los helpers asumen un solo tenant).
- Los cambios de rol requieren auditoría externa (tabla `role_audit` + trigger definidos en F-S00-05).

## Convenciones para nuevos recursos

Al añadir una tabla o bucket nuevo al dominio:

1. Añadir una fila en la sección correspondiente de este ADR.
2. Crear la migración SQL con las policies (nombre: `<tabla>_<accion>_<rol>`).
3. Añadir tests pgTAP en `supabase/tests/rls/<tabla>.sql` (≥ 1 positivo + 1 negativo por policy).
4. Abrir un PR con ADR + migración + tests juntos. No se aceptan policies sin tests ni tests sin policy documentada.
